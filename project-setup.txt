# 🔷 PART 1 — SYSTEM ARCHITECTURE, TECHNOLOGY STACK, AND FOUNDATIONAL RULES

---

## 1.1 Project Objective

Build a full-stack, production-grade Event Management Platform where:

* Colleges host events
* Organisers manage events
* Participants register and attend events
* Attendance is verified using QR codes
* Email notifications are automated
* Role-based dashboards control system access

The system must be scalable, secure, modular, and production-ready.

This is NOT a prototype.
This is not a hackathon demo.
This must follow real-world backend architecture standards.

---

## 1.2 Core Technology Stack (Non-Negotiable)

### Frontend & Backend Framework

* **Next.js (App Router)**
* TypeScript (strict mode enabled)

Use Next.js as:

* Frontend (React Server + Client components)
* Backend (API route handlers)

Do NOT separate backend into another Express server.

---

### Database Layer

* PostgreSQL (hosted database)
* Prisma ORM

All database interactions MUST go through Prisma.

No raw SQL unless absolutely necessary.

---

### Authentication

* JWT-based authentication
* Password hashing using bcrypt
* HTTP-only cookies for session storage

No session storage in localStorage.
No plaintext password storage.
No insecure token exposure.

---

### Email Service

* Nodemailer
* SMTP-based configuration

Emails must be sent from backend only.

---

### Image Storage

Images (profile photos, event banners) must:

* Be uploaded to cloud storage (e.g., Cloudinary or S3)
* Store ONLY the image URL in the database
* NEVER store binary image data in PostgreSQL

---

## 1.3 High-Level System Architecture

The application must follow a layered structure:

```
Client (UI)
   ↓
API Route Handlers
   ↓
Service Layer (business logic)
   ↓
Prisma ORM
   ↓
PostgreSQL Database
```

### Strict Rule:

Business logic must NOT exist directly inside route handlers.

Instead:

* Route handler validates request
* Calls service function
* Service function interacts with database

This ensures maintainability and scalability.

---

## 1.4 Role-Based Access Control (RBAC)

The system has 4 roles:

* SUPERADMIN
* ADMIN
* ORGANISER
* USER

Each user must have exactly one role.

The role must:

* Be stored in database
* Be embedded inside JWT
* Be validated on every protected API call

Frontend role rendering is not security.
Backend validation is mandatory.

---

## 1.5 Dashboard Philosophy

There must be only ONE dashboard route:

```
/dashboard
```

The dashboard UI must conditionally render based on user role.

Do NOT create:

```
/admin-dashboard
/user-dashboard
/organiser-dashboard
```

All logic must branch inside the same layout.

This ensures:

* Clean routing
* Maintainability
* Centralized authentication

## 1.6 Environment Configuration

The system must use environment variables for:

* DATABASE_URL
* JWT_SECRET
* SMTP_HOST
* SMTP_PORT
* SMTP_USER
* SMTP_PASS
* CLOUD_STORAGE_KEYS

Never hardcode credentials.

Include `.env.example` file for reference.

## 1.7 Security Architecture Rules

1. All protected routes must use middleware.
2. Every API route must verify JWT.
3. Role must be validated server-side.
4. Inputs must be validated (using Zod or similar).
5. UUIDs must be used for tokens.
6. Passwords must be hashed.
7. QR tokens must not be predictable.
8. Never trust frontend state.

---

## 1.8 Database Design Philosophy

* Use relational integrity
* Use foreign keys
* Use cascading deletes where appropriate
* Avoid duplicate data
* Use indexes for frequently queried fields

Relations that must exist:

* User → College
* Event → Organiser (User)
* Event → College
* Registration → User
* Registration → Event
* Bookmark → User
* Bookmark → Event
* Notification → User

# 🔷 PART 2 — AUTHENTICATION, AUTHORIZATION & ROLE SYSTEM (DETAILED SPECIFICATION)

---

# 2.1 Authentication Philosophy

Authentication must be:

* Stateless (JWT-based)
* Secure
* Server-validated
* Role-aware
* Production-ready

The system must **never trust frontend state**.

All protected access must be validated server-side.

---

# 2.2 Authentication Method

The system must use:

* Email + Password login
* bcrypt password hashing
* JWT (JSON Web Token)
* HTTP-only secure cookies for session storage

Do NOT:

* Store tokens in localStorage
* Store passwords unhashed
* Store session in memory

---

# 2.3 User Roles (Strict Enum)

Roles must be defined in Prisma schema as:

```
enum Role {
  SUPERADMIN
  ADMIN
  ORGANISER
  USER
}
```

Each user must have exactly one role.

No multi-role users.
No dynamic roles stored as strings.

---

# 2.4 Registration Types

There are TWO types of account creation:

### 1️⃣ Self Registration (USER only)

When a normal participant signs up:

* Role must automatically be USER
* They cannot choose their role
* They cannot assign themselves ORGANISER or ADMIN

---

### 2️⃣ Admin-Created Accounts

Only:

* SUPERADMIN can create ADMIN accounts
* ADMIN can create ORGANISER accounts

Users cannot create elevated accounts.

This must be enforced in backend.

---

# 2.5 Password Security

On account creation:

1. Validate password strength:

   * Minimum 8 characters
   * Must include letters and numbers

2. Hash password using bcrypt:

   * Salt rounds: 10 or higher

3. Store only hashed password

Never log password.
Never return password in API response.

---

# 2.6 Login Flow (Detailed Backend Logic)

When user attempts login:

### Step 1: Validate input

* Email format
* Password not empty

### Step 2: Find user by email

If not found:

* Return 401 (Invalid credentials)

### Step 3: Compare password using bcrypt.compare()

If invalid:

* Return 401

### Step 4: Generate JWT

JWT payload must include:

```
{
  userId: string,
  role: Role,
  email: string
}
```

Do NOT include:

* Password
* Sensitive internal fields

---

### Step 5: Set HTTP-only cookie

Cookie settings:

* httpOnly: true
* secure: true (in production)
* sameSite: strict
* maxAge: 7 days

---

### Step 6: Return sanitized user object

Return:

```
{
  id,
  name,
  email,
  role,
  profileImageUrl
}
```

---

# 2.7 JWT Configuration

JWT must:

* Be signed with JWT_SECRET
* Have expiration (7 days recommended)
* Use HS256 algorithm

If token expired:

* Force re-login

---

# 2.8 Middleware Protection

Create global middleware:

```
/middleware.ts
```

Middleware must:

1. Check if route is protected
2. Extract JWT from cookie
3. Verify token
4. Attach decoded user to request
5. Redirect to login if invalid

---

Protected routes:

```
/dashboard/*
/api/*
(except auth routes)
```

---

# 2.9 Role-Based Route Authorization

Authentication ≠ Authorization.

Even if logged in,
you must check role.

Example:

If route is:

```
POST /api/admin/create-organiser
```

Backend must verify:

```
if (user.role !== ADMIN) {
   return 403
}
```

Frontend hiding button is NOT security.

---

# 2.10 Current User Retrieval

Create helper function:

```
getCurrentUser()
```

This function must:

1. Extract token from cookie
2. Verify token
3. Fetch fresh user from DB
4. Return full user object

This prevents stale token problems.

---

# 2.11 Logout Flow

When user logs out:

1. Clear cookie
2. Invalidate session client-side
3. Redirect to login

No token blacklisting required (stateless system).

---

# 2.12 Admin Account Creation Flow

When SUPERADMIN creates ADMIN:

1. Validate SUPERADMIN role
2. Create user with role ADMIN
3. Generate temporary password
4. Hash password
5. Store in DB
6. Send account email
7. Force password change on first login (optional improvement)

---

When ADMIN creates ORGANISER:

Same logic,
but role must be ORGANISER.

---

# 2.13 Security Edge Cases to Handle

Must prevent:

* Token tampering
* Privilege escalation
* Role injection via frontend
* Direct API calls from Postman without token
* Accessing dashboard without login

All must return proper 401 or 403.

---

# 2.14 Password Reset (Basic Structure Only)

Even if not fully implemented now,
structure must allow future:

* Request reset token
* Email reset link
* Token expiration (15 min)
* Hash reset token in DB

Design schema extensible for this.

---

# 2.15 API Endpoints Required (Authentication)

Must create:

```
POST /api/auth/register
POST /api/auth/login
POST /api/auth/logout
GET  /api/auth/me
```

---

### /register

* Only creates USER
* Hash password
* Save profile photo URL

---

### /login

* Validates credentials
* Issues JWT cookie

---

### /logout

* Clears cookie

---

### /me

* Returns current authenticated user
* Must verify token

---

# 2.16 Data Validation

Use Zod or similar.

Validate:

* Email format
* Password strength
* Role assignment logic
* Required fields

Never trust request body directly.

---

# 2.17 Database Indexing for Auth

Ensure:

* Email is UNIQUE
* Email indexed

This prevents duplicate accounts and speeds login lookup.

---

# 2.18 Failure Response Standards

Authentication errors:

* 400 → Validation error
* 401 → Invalid credentials
* 403 → Forbidden (wrong role)
* 500 → Internal server error

Never expose:

* Database errors
* Stack traces
* Internal queries

# 3.4 MODEL: College

Purpose:
Represents a college/institution under which events and users exist.

Structure:

* id → UUID (primary key)
* name → unique
* createdAt
* updatedAt

Relationships:

* One College has many Users
* One College has many Events

Constraints:

* College name must be unique
* Indexed for fast lookup

Deleting a college must NOT automatically delete users.

# 3.5 MODEL: User

Fields:

* id → UUID (primary key)
* name
* email → unique + indexed

* role → Role enum
* profileImageUrl → string (NOT nullable, mandatory)
* collegeId → foreign key
* createdAt
* updatedAt

Relations:

* Belongs to one College
* Has many Events (if ORGANISER)
* Has many Registrations
* Has many Notifications
* Has many Bookmarks

Critical Constraints:

* Email must be unique
* profileImageUrl must not be null
* role default must be USER
* password must never be selected in public queries

---

# 3.6 MODEL: Event

Represents an event created by an organiser.

Fields:

* id → UUID
* title
* description
* bannerUrl (string)
* date (DateTime)
* venue
* capacity (Int)
* isPaid (Boolean)
* organiserId (FK → User)
* collegeId (FK → College)
* createdAt
* updatedAt

Relations:

* Belongs to one organiser (User)
* Belongs to one college
* Has many registrations
* Has many bookmarks

Critical Rules:

* organiserId must reference a user with role ORGANISER
* capacity must be positive
* date must be future date at creation time

Indexes required:

* Index on date
* Index on organiserId
* Index on collegeId

---

# 3.7 MODEL: Registration

This connects User and Event.

This is a junction table with additional metadata.

Fields:

* id → UUID
* userId → FK
* eventId → FK
* qrToken → unique string
* attendanceMarked → Boolean (default false)
* status → RegistrationStatus
* createdAt
* updatedAt

Relations:

* Belongs to one user
* Belongs to one event

Critical Constraints:

* Composite unique constraint:
  (userId, eventId)

This prevents double registration.

* qrToken must be unique
* attendanceMarked default false

Cascade Rules:

If event is deleted → delete registrations
If user is deleted → delete registrations

---

# 3.8 MODEL: Notification

Fields:

* title

Relations:

* Belongs to one user

Indexes:

* Index on userId

# 3.9 MODEL: Bookmark

Fields:

* createdAt

Relations:

* Belongs to one user

Composite unique:
(userId, eventId)

Prevents duplicate bookmarks.

Cascade:
Delete event → delete bookmarks
Delete user → delete bookmarks

If not:
Return 404.

Reject with 400 (Event expired).

Count current confirmed registrations.

Reject with 400 (Event full).

Insert:

* userId
* eventId
* qrToken
* status = CONFIRMED
* attendanceMarked = false

---

### Step 8: Create Notification

Insert:

Title:

```
You're Registered 🎉
```

Message:
Include event name and date.

---

### Step 9: Trigger Email

Send confirmation email (defined in future part).

---

### Step 10: Return Success Response

Return:

* registrationId
* qrToken
* event details

# 3️⃣ QR Code Design (Security Model)

DO NOT store plain session ID in QR.

Instead generate a signed payload.

### QR Payload Structure (JSON)

```
{
  "sessionId": "uuid",
  "exp": 1700000000,
  "nonce": "randomString",
  "signature": "HMAC-SHA256"
}
```

Where:

* exp = expiration timestamp (unix seconds)
* nonce = random unique value
* signature = HMAC(secretKey, sessionId + exp + nonce)

Secret key stored in server environment variable.

Example:

```
ATTENDANCE_QR_SECRET=super-long-random-key
```

# 4️⃣ QR Generation Logic (Server Side)

When admin clicks "Start Session":

1. Update session:

   * status = ACTIVE
   * startTime = now()
   * qrExpiresAt = now() + 10 minutes
   * qrSecret = random 64-byte string

2. Create payload:

```
data = sessionId + expiration + nonce
signature = HMAC_SHA256(data, ENV_SECRET)
```

3. Create JSON payload
4. Convert to Base64
5. Generate QR using library

Recommended library for Next.js:

```
qrcode
```

Example:

```ts
import QRCode from 'qrcode'

const qrDataUrl = await QRCode.toDataURL(payloadString)
```

Return to frontend.

# 6️⃣ Attendance Validation Algorithm (CRITICAL SECTION)

Backend must:

### Step 1: Decode payload

Parse JSON safely.
Reject if malformed.

---

### Step 2: Validate expiration

```
if (currentTime > payload.exp) → reject
```

Return:

```
QR code expired
```

---

### Step 3: Validate signature

Recreate:

```
expectedSignature = HMAC(sessionId + exp + nonce, ENV_SECRET)
```

Compare with payload.signature.

If mismatch → reject (possible tampering).

---

### Step 4: Fetch session

Find session by sessionId.

Reject if:

* Not found
* status != ACTIVE
* now() > session.endTime

---

### Step 5: Validate student enrollment

Check Enrollment table:

```
where studentId = currentUser.id
and courseId = session.courseId
```

If not found → reject.

---

### Step 6: Prevent duplicate attendance

Before insert:

Check:

```
Attendance.findUnique({
  where: {
    sessionId_studentId: {
      sessionId,
      studentId
    }
  }
})
```

If exists → return:

"Already marked present"

---

### Step 7: Insert Attendance

Create record:

* sessionId
* studentId
* scannedAt = now()
* ipAddress (optional)
* userAgent (optional)

---

### Step 8: Return success

Response:

```
{
  success: true,
  message: "Attendance marked successfully"
}
```
